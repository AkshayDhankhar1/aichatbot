# CRM RAG Chatbot

A full-stack Retrieval-Augmented Generation chatbot over a read-only `docs/`
folder. Hybrid (dense + BM25) retrieval on Pinecone, grounded generation with
Groq (Llama), and a Vite + React chat UI that renders inline charts on request.

- **Backend:** FastAPI + LangChain
- **Frontend:** Vite + React (JavaScript)
- **LLM:** Groq (`langchain-groq`) — low temperature, strictly grounded
- **Embeddings:** local `sentence-transformers/all-MiniLM-L6-v2` (**384 dims**)
- **Vector DB:** Pinecone serverless, **dimension 384, metric `dotproduct`**
- **Search:** hybrid dense + sparse BM25 (`pinecone-text`), alpha-weighted fusion
- **Charts:** backend detects explicit chart requests and returns structured
  data; frontend renders with `recharts`
- **Persistence:** none — session-only chat state (no database)

---

## Design decisions (read me)

- **Dimensions.** The original brief asked for 128-dim embeddings *and* a 384-dim
  index — these are mutually incompatible (dense vectors must match the index
  dimension). Per your choice we use the model's **native 384 dims** and a
  **384-dim index**. No PCA/projection layer.
- **No reranker.** Per your choice, there is no cross-encoder rerank step (it
  would pull in a second heavy `torch` model). The **anti-hallucination gate** is
  instead the hybrid **fusion score** of the top retrieved chunk vs.
  `MIN_SCORE_THRESHOLD` — below threshold, the bot answers "not in the
  documents" instead of guessing.
- **No Docker.** Deploy via native buildpacks (Procfile).
- **Tabular data caveat.** `docs/` is mostly tabular CRM data. With plain
  chunking (your choice) and no reranker, the bot answers reliably when a figure
  is *written* in a document (e.g. totals stated in `DATA_DICTIONARY.md`), but it
  will not reliably compute multi-row sums itself. The system prompt tells it to
  refuse rather than fabricate such numbers.
- **Read-only `docs/`.** Only `ingest.py` reads `docs/`, and it never writes,
  edits, or deletes anything there. The runtime chatbot only queries Pinecone.

## Project layout

```
docs/                     # your source files (READ-ONLY) — CRM sample data
backend/
  app/
    main.py               # FastAPI app + CORS
    config.py             # env config (pydantic-settings)
    prompts.py            # system prompts + anti-hallucination logic (tune here)
    loaders.py            # multi-format loaders (csv/tsv/json/xlsx/docx/pdf/md/txt)
    vectorstore.py        # embeddings + BM25 + Pinecone helpers
    ingest.py             # standalone: docs -> chunk -> embed -> upsert
    rag_chain.py          # hybrid retrieve -> gate -> generate (+ chart mode)

    routes/health.py      # GET /health
    data/bm25_params.json # generated by ingest; committed for deploy

  .env / .env.example
frontend/
  src/
    App.jsx               # chat state + composer
    api.js                # SSE client (reads VITE_API_BASE_URL)
    components/Message.jsx
    components/ChartMessage.jsx   # recharts
  .env / .env.example
```

## 1. Prerequisites & keys

- Python 3.11+ and Node 18+
- **Groq API key** — https://console.groq.com (free tier)
- **Pinecone API key** — https://app.pinecone.io (free tier)

Fill in `backend/.env` (copied from `.env.example`):

```
GROQ_API_KEY=...            # required
PINECONE_API_KEY=...        # required
PINECONE_INDEX_NAME=crm-rag
PINECONE_CLOUD=aws
PINECONE_REGION=us-east-1
```

You do **not** need to pre-create the Pinecone index — `ingest.py` creates it
(dim 384, metric `dotproduct`) if it doesn't exist.

## 2. Run the backend locally

```bash
cd backend
python -m venv .venv
# Windows:  .venv\Scripts\activate      | macOS/Linux: source .venv/bin/activate
pip install -r requirements.txt
```

> First install pulls `torch` via `sentence-transformers` (a few hundred MB).
> CPU-only machines can slim it: `pip install torch --index-url https://download.pytorch.org/whl/cpu`

### Ingest the documents (run once, and whenever docs change)

```bash
python -m app.ingest            # add --recreate to wipe & re-upsert
```

This reads `../docs/`, chunks, fits BM25, embeds, and upserts to Pinecone. It
also writes `app/data/bm25_params.json` (commit this so the deployed backend uses
the same sparse statistics).

### Start the API

```bash
uvicorn app.main:app --reload --port 8000
```

Check: http://localhost:8000/health → `{"status":"ok"}` · Swagger at `/docs`.

---

## 3. Run the frontend locally

```bash
cd frontend
npm install
npm run dev        # http://localhost:5173
```

`frontend/.env` points the UI at the backend:
```
VITE_API_BASE_URL=http://localhost:8000
```

Try: *"What was total closed-won revenue?"* and *"plot deals by stage as a bar chart"*.

## How ingestion works & adding new docs

1. Drop files into `docs/` (supported: `.pdf .docx .xlsx .csv .tsv .json .md .txt`).
   Junk like `__MACOSX/` and `._*`/dotfiles is skipped automatically.
2. Re-run `python -m app.ingest` (use `--recreate` if you removed/renamed files
   and want a clean index).
3. Commit the regenerated `backend/app/data/bm25_params.json`.
4. Redeploy the backend (below) — no code change needed.

Tuning knobs live in `backend/.env` (`HYBRID_ALPHA`, `TOP_K`, `TOP_N`,
`MIN_SCORE_THRESHOLD`) and grounding text lives in `backend/app/prompts.py`.

## 4. Deployment

### Backend → Railway (native buildpack, recommended)

Railway gives more RAM headroom than Render's free tier, which matters because
`torch` + the embedding model are memory-hungry (**flag: the 512 MB free tiers
can OOM; use Railway's Hobby plan or a paid Render instance if you see restarts**).

1. Push this repo to GitHub (ensure `backend/app/data/bm25_params.json` is
   committed and `.env` is **not**).
2. Go to https://railway.app → **New Project → Deploy from GitHub repo**.
3. In the service **Settings**:
   - **Root Directory:** `backend`
   - Railway auto-detects Python and uses the `Procfile`
     (`uvicorn app.main:app --host 0.0.0.0 --port $PORT`). No start command needed.
4. **Variables** tab — add each key:
   - `GROQ_API_KEY`
   - `GROQ_MODEL` = `llama-3.3-70b-versatile`
   - `PINECONE_API_KEY`
   - `PINECONE_INDEX_NAME` = `crm-rag`
   - `PINECONE_CLOUD` = `aws`, `PINECONE_REGION` = `us-east-1`
   - `EMBEDDING_MODEL_NAME` = `sentence-transformers/all-MiniLM-L6-v2`
   - `HYBRID_ALPHA` `TOP_K` `TOP_N` `MIN_SCORE_THRESHOLD` (optional; defaults exist)
   - `ALLOWED_ORIGINS` = `http://localhost:5173` for now (add the Vercel URL after step below)
5. **Networking → Generate Domain**. Copy the URL, e.g.
   `https://your-app.up.railway.app`. Verify `/health`.

> Ingestion is a one-off you run **locally** against your Pinecone index; you do
> not run it on Railway. The deployed backend only queries Pinecone.

**Render alternative:** New → Web Service → root dir `backend`, Build
`pip install -r requirements.txt`, Start `uvicorn app.main:app --host 0.0.0.0 --port $PORT`,
add the same env vars. Use a paid instance type to avoid OOM.

### Frontend → Vercel

1. https://vercel.com → **Add New → Project** → import the repo.
2. **Root Directory:** `frontend`. Framework preset: **Vite** (auto). Build
   `npm run build`, output `dist` (auto).
3. **Environment Variables:** add `VITE_API_BASE_URL` = your Railway URL
   (e.g. `https://your-app.up.railway.app`). **Deploy.**
4. Copy the Vercel URL (e.g. `https://your-app.vercel.app`).

### Wire CORS

Back in Railway **Variables**, set:
```
ALLOWED_ORIGINS=http://localhost:5173,https://your-app.vercel.app
```
Redeploy the backend. Done.

## Redeploying after changes

- **Docs changed:** re-run `python -m app.ingest` locally → commit
  `bm25_params.json` → push. Railway auto-redeploys on push.
- **Backend code changed:** push → Railway redeploys. Update env vars in the
  dashboard if needed.
- **Frontend changed:** push → Vercel redeploys. Change `VITE_API_BASE_URL` in
  Vercel if the backend URL changed.

## API

`POST /chat` — request `{ "message": string, "history": [{role, content}] }`,
response `text/event-stream` of SSE frames:
`{"type":"token","content":"..."}`, then
`{"type":"done","sources":[...],"chart": {...}|null}` (or `{"type":"error","message":"..."}`).
